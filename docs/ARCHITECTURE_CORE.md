# NodeOrchestrator 架构核心

> 技术栈见 [spec/TECH_STACK.md](spec/TECH_STACK.md)

## 定位

基于节点的 Pi Agent 编排器。每个节点是一个独立 Pi RPC 子进程，节点通过数据流边连接形成可执行图。图结构在执行过程中由 LLM 动态生成。

## 核心概念

### Node（节点）

节点是基本执行单元，每个节点对应一个正在运行的 `pi --mode rpc` 子进程。节点有输入/输出端口，通过边接收和发送数据。节点有状态、重试计数和升级历史。

### Edge（边）

边连接节点的输出端口到另一节点的输入端口，数据在边上流动。边可以带条件，只有条件满足时才传递数据。

### Group（任务组）

任务组是节点的容器，对应一个独立任务工作区域。组有输入/输出端口，对外表现为黑盒。**一个任务组内的所有执行节点共享同一个 Git worktree**，组内节点协同修改同一份代码。

### Graph（图）

图是完整的可执行工作流，包含节点、边和任务组。图结构在执行过程中由 LLM 动态生成（如 clarification 节点、planning 生成的任务组）。

## 节点类型

| 类型 | 职责 | 是否等用户 | 默认强度 | 说明 |
|------|------|-----------|---------|------|
| `requirement` | 接收用户初始需求 | ✅ | 高 | 纯前端输入，触发 LLM 分析 |
| `analysis` | LLM 分析需求 | ❌ | 高 | 分析需求是否明确，输出分析报告 |
| `clarification` | 生成澄清问题 | ✅ | **高** | 执行到一半暂停，等用户选择/输入 |
| `confirmation` | 确认最终需求 | ✅ | **高** | 展示需求摘要，等用户确认或修改 |
| `planning` | 生成执行子图 | ❌ | 高 | 确认需求后，LLM 生成完整任务组 |
| `coder` | 编码实现 | ❌ | 低 | 接收指令，输出代码 diff |
| `reviewer` | 代码审查 | ❌ | 中 | 接收 diff，输出审查意见 |
| `build` | 编译验证 | ❌ | 低 | 执行构建/测试，输出结果 |
| `browser` | 浏览器验证 | ❌ | 中 | SPA 交互、截图验证 |
| `audit` | 多角度审核 | ❌（默认自动） | 中 | 安全性/性能/质量等多维度审核 |
| `coordinator` | 智能指导 | ❌ | 高 | 质量循环耗尽时介入分析，输出指导建议 |
| `create_pr` | 创建 PR | ❌ | 低 | 任务组完成后自动创建并推送 PR |
| `pr_review` | 审核 PR | ❌ | 中 | LLM 自动审核 PR 的 diff，输出审核意见 |
| `merge_pr` | 合并 PR | ❌ | 低 | pr_review 通过后自动合并 |
| `completion` | 任务完成 | ❌ | 低 | PR 合并后汇总产物，标记完成 |

**强度档位**：低 / 中 / 高，系统根据档位自动选择对应模型和思考强度。
- 低：低成本模型，无扩展思考
- 中：中等模型，按需扩展思考
- 高：最强模型，启用扩展思考
- 需求相关节点（clarification、confirmation）默认高强度，确保需求理解准确

## 三阶段执行模型

### 阶段1：需求工程（交互式）

```
[requirement] → LLM分析 → 不够明确?
                              ↓
                    [clarification] ←── 用户输入 ──┐
                              ↓                    │
                    继续分析 ──┘                    │
                              ↓                    │
                    明确后 → [confirmation] ──→ 用户确认
                                                  ↓
                                         同步主分支（git pull）
```

- 用户深度参与，每步交互
- LLM 动态生成 clarification 节点，可多轮
- 用户可随时修改 requirement，重新分析
- **confirmation 确认后，自动同步主分支**，确保后续工作基于最新代码

### 阶段2：执行工程（全自动）

```
[confirmation] ──→ [planning] ──→ 📦 任务组（自动生成）
                                      │
                    ┌─────────────────┼─────────────────┐
                    ↓                 ↓                 ↓
              ┌─────────┐      ┌─────────┐      ┌─────────┐
              │ 分支1   │      │ 分支2   │      │ 分支N   │
              │ [coder] │      │ [coder] │      │ [coder] │
              │   ↓     │      │   ↓     │      │   ↓     │
              │[reviewer]│     │[reviewer]│     │[reviewer]│
              │   ↓     │      │   ↓     │      │   ↓     │
              │ [build] │      │ [build] │      │ [build] │
              └────┬────┘      └────┬────┘      └────┬────┘
                   └─────────────────┼─────────────────┘
                                    ↓
                              [多角度审核]
                              （安全/性能/质量）
                                    ↓
                              📦 任务组完成
```

- planning 节点一次性生成完整任务组（含可并行分支）
- 任务组基于同步后的最新主分支创建 worktree，确保代码最新
- 任务组内固定模式：coder → reviewer → build
- 组内循环（不通过回 coder）不扩散到外层
- 全自动运行，用户仅旁观状态

### 阶段3：PR 与完成

```
📦 任务组完成 ──→ [audit] ──→ 通过 ──→ [create_pr] ──→ PR 推送
                    │                              │
                    └── 不通过 ──→ 回到分支修改     │
                                                   ↓
                                             [pr_review]
                                                   │
                                              通过 → [merge_pr] ──→ [completion]
                                                   │
                                              不通过 ──→ 回到分支修改
```

- 任务组完成后先经过自动 audit 审核
- audit 通过 → 自动创建并推送 PR
- LLM 自动审核 PR（pr_review 节点读取 PR diff 进行审查）
- pr_review 通过 → 自动合并 PR → 标记 completion
- audit 或 pr_review 不通过 → 回到对应分支继续修改（质量循环）

## 固定执行模式（任务组内）

每个执行分支遵循以下固定模式：

```
[coder] ──diff──→ [reviewer]
   ↑                  │
   │              通过 → [build]
   │                  │    │
   │              失败 ←───┘    通过
   │                             ↓
   └──────────────────── 不通过 ←─┘
```

- **coder** → **reviewer**：coder 输出 diff，reviewer 审查
- **reviewer** → **build**：reviewer 通过，进入 build 验证
- **build** → **coder**：build 失败，带错误信息回到 coder
- **reviewer** → **coder**：reviewer 不通过，带意见回到 coder

## 节点状态机

```
Idle → Ready → Running → Completed ──→ 触发下游
```

LLM 节点**不会失败**——它总会产出结果。问题是结果质量是否达标。

### 质量循环（任务组内）

```
[coder] ──diff──→ [reviewer]
   ↑                  │
   │              通过 → [build]
   │                  │    │
   │              失败 ←───┘    通过
   │                             ↓
   └──────────────────── 不通过 ←─┘
        （循环次数 +1）
```

- 每完成一次循环（coder→reviewer→coder 或 coder→reviewer→build→coder），循环计数 +1
- 循环次数达到阈值（如 3 次）仍不通过 → 触发指导/升级流程

### 指导与升级流程

```
循环次数 >= max
        │
        ├──→ [coordinator 节点介入]
        │       │
        │       ├──→ 输出指导建议（修改方案、调整范围等）
        │       │
        │       └──→ 判断是否需要升级模型
        │               │
        │               ├──→ 建议升级 ──→ set_model ──→ 继续循环
        │               │
        │               └──→ 建议人工介入 ──→ WaitingForUser
        │                       ↓
        │               用户决策：继续 / 修改需求 / 中止
        │
        └──→ 无更高模型且 coordinator 无法解决
                ↓
        WaitingForUser（用户必须决策）
```

### 各类型节点 WaitingForUser 触发条件

| 节点类型 | 是否可能等待用户 | 触发条件 |
|---------|---------------|---------|
| requirement | ✅ | 等待用户输入初始需求 |
| clarification | ✅ | 等待用户选择/输入澄清答案 |
| confirmation | ✅ | 等待用户确认或修改 |
| coder/reviewer/build/browser | ❌（正常情况下） | — |
| 同左 | ✅（质量循环耗尽） | 循环次数达到上限且 coordinator 无法解决 |
| audit | ❌（默认自动） | 自动通过或自动回退 |

## 升级与指导策略

节点执行不会失败，但产出质量可能不达标。处理流程：

```
质量循环（coder→reviewer→build→coder）
        │
        ├──→ 审核通过 / 构建成功 → 继续下游
        │
        └──→ 审核不通过 / 构建失败 → 循环计数 +1
                │
                ├──→ 循环次数 < max（如 3） → 继续循环
                │
                └──→ 循环次数 >= max
                        │
                        ├──→ [coordinator 节点介入]
                        │       │
                        │       ├──→ 输出指导建议
                        │       │       （修改方案、调整范围、换思路等）
                        │       │
                        │       └──→ 判断是否需要升级模型
                        │               │
                        │               ├──→ 建议升级 → 提升强度档位（低→中→高）→ 继续循环
                        │               │
                        │               └──→ 建议人工介入 → WaitingForUser
                        │                       ↓
                        │               用户决策：继续 / 修改需求 / 中止
                        │
                        └──→ 无更高模型且 coordinator 无法解决
                                ↓
                        WaitingForUser（用户必须决策）
```

- **循环**：同模型下多次修改（coder 根据 reviewer/build 反馈继续改）
- **指导**：coordinator 分析循环卡住的根因，输出结构化建议
- **升级**：set_model 换更强模型，循环计数重置
- **用户介入**：最终兜底，用户可修改需求、调整范围或中止任务

## 数据流

节点间传递的是带类型的消息，主要类型包括：instruction（任务指令）、code（代码）、diff（代码变更）、review（审查意见）、artifact（产物）、error（错误信息）、approval（审核结果）。消息包含来源节点、端口和时序信息。

## 任务组调度

任务组有**独立内部调度器**：

1. 组被激活时，内部调度器启动
2. 组内节点状态独立管理，与外层图隔离
3. 组内循环只在组内流转（coder→reviewer→coder）
4. 组级输入数据流入组内对应节点
5. 组内输出节点完成 → 组标记 Completed → 触发外层下游
6. 组内节点失败无法恢复 → 组标记 WaitingForUser → 外层图暂停

## 并行执行

planning 节点生成的任务组可包含多个并行分支：

- 分支间无依赖时并行执行
- 分支间有依赖时按拓扑顺序
- 各分支独立重试/升级
- 所有分支完成 + 审核通过 → 任务组完成

## 安全

- 工具白名单 per Profile
- Git worktree 隔离 per 任务组（组内节点共享同一份代码）
- 组级预算限制（token / cost）
- Secrets 分域，生产凭据不下发 Worker
- 审计：RPC 事件、模型切换、费用、artifacts

## 前端交互设计

### 页面结构

| 页面 | 功能 |
|------|------|
| **项目列表** | 展示项目卡片，支持添加项目（GitHub 链接） |
| **节点图** | 三栏布局：任务图列表（左）+ 节点画布（中）+ 执行详情（右） |
| **设置** | 配置 Pi Agent 模型、全局参数 |

### 项目列表页

- 空状态提示添加项目
- 添加项目：输入 GitHub 仓库链接，系统克隆并索引
- 项目卡片展示：项目名称、最近任务状态、快速进入按钮

### 节点图页

**左侧栏：任务图列表**
- 可折叠/展开
- 列出当前项目的所有任务图
- 支持新建任务图

**中间：节点画布**
- React Flow 渲染节点和边
- **画布整体可拖动（平移），单节点不可拖动**——节点位置由系统自动布局，避免用户拖拽导致混乱
- 节点实时显示状态颜色（空闲/运行中/等待用户/完成）
- **节点上展示当前使用的模型和思考强度**（如 Sonnet 4.6 / 扩展思考）
- 点击节点 → 右侧栏显示该节点执行详情
- 左上角返回按钮 → 回到项目列表

**右侧栏：执行详情**
- 点击节点后展开
- 展示该 Pi Agent 的完整对话历史
- 实时显示新的消息、token 消耗
- 交互式节点（clarification/confirmation）在此输入

### 设置页

- **模型强度映射**：低/中/高 分别映射到哪个具体模型 + 是否启用扩展思考
- 节点默认强度：可按节点类型覆盖默认档位
- 全局预算限制
- Git 配置、GitHub Token
- 工具白名单默认设置

## MVP 阶段

1. **需求工程**：requirement → analysis → clarification → confirmation
2. **执行工程**：confirmation → planning → 任务组（单分支）
3. **固定模式**：coder → reviewer → build → audit → create_pr → pr_review → merge_pr → completion
4. **升级策略**：质量循环 → coordinator 指导 → set_model → 用户决策
5. **任务组**：组内循环、组级状态、组对外黑盒
6. **并行分支**：planning 生成多分支任务组
7. **前端**：项目列表 → 节点图（三栏布局）→ 设置页
